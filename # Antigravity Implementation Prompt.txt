## TASK 3 ‚Äî BLOCCANTE: Schedulare `expire-sessions`

**Problema:** `supabase/functions/expire-sessions/index.ts` esiste ma non √® schedulata.

**Azioni:**

### Opzione A ‚Äî Supabase Cron (consigliata)
Nel dashboard Supabase ‚Üí Database ‚Üí Extensions ‚Üí abilitare `pg_cron`
Poi in SQL Editor:
```sql
SELECT cron.schedule(
  'expire-sessions-hourly',
  '0 * * * *',  -- ogni ora
  $$
  SELECT net.http_post(
    url := current_setting('app.supabase_url') || '/functions/v1/expire-sessions',
    headers := jsonb_build_object(
      'Authorization', 'Bearer ' || current_setting('app.service_role_key'),
      'Content-Type', 'application/json'
    ),
    body := '{}'::jsonb
  );
  $$
);
```

### Opzione B ‚Äî Netlify Scheduled Function
Crea `netlify/functions/expire-sessions-cron.ts`:
```typescript
import type { Config } from '@netlify/functions'

export default async function handler() {
  const res = await fetch(`${process.env.SUPABASE_URL}/functions/v1/expire-sessions`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${process.env.SUPABASE_SERVICE_ROLE_KEY}` },
  })
  return { statusCode: res.ok ? 200 : 500 }
}

export const config: Config = { schedule: '@hourly' }
```

Aggiungere in `.env.local` (e in Netlify env vars):
```
SUPABASE_SERVICE_ROLE_KEY=...
```

## TASK 5 ‚Äî IMPORTANTE: Rate limiting codici pairing

**Problema:** `generate_pairing_code()` in `supabase/migrations/003_functions.sql` non ha rate limiting.

**Azione ‚Äî Nuova migration SQL:**

Crea `supabase/migrations/008_rate_limiting.sql`:
```sql
-- Tabella tracking tentativi per user
CREATE TABLE IF NOT EXISTS public.pairing_rate_limits (
  user_id     TEXT NOT NULL,
  window_start TIMESTAMPTZ NOT NULL DEFAULT date_trunc('minute', NOW()),
  attempt_count INT NOT NULL DEFAULT 1,
  PRIMARY KEY (user_id, window_start)
);

-- Funzione con rate limit integrato
CREATE OR REPLACE FUNCTION public.generate_pairing_code_rate_limited()
RETURNS TEXT LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
DECLARE
  v_uid TEXT := requesting_user_id();
  v_window TIMESTAMPTZ := date_trunc('minute', NOW());
  v_count INT;
  v_code TEXT;
BEGIN
  -- Max 5 generazioni per minuto per utente
  INSERT INTO pairing_rate_limits (user_id, window_start, attempt_count)
  VALUES (v_uid, v_window, 1)
  ON CONFLICT (user_id, window_start)
  DO UPDATE SET attempt_count = pairing_rate_limits.attempt_count + 1
  RETURNING attempt_count INTO v_count;

  IF v_count > 5 THEN
    RAISE EXCEPTION 'Troppi tentativi. Attendi un minuto.';
  END IF;

  RETURN generate_pairing_code();
END; $$;

GRANT EXECUTE ON FUNCTION public.generate_pairing_code_rate_limited() TO authenticated;
```

Aggiornare `src/pages/pairing/ShowQRPage.tsx` per chiamare la nuova funzione via RPC.

## TASK 6 ‚Äî IMPORTANTE: Retry logic per confirm/revoke

**Problema:** `src/hooks/useSession.ts` ‚Äî le funzioni `confirmConsent()` e `revokeConsent()` non hanno retry in caso di errore rete transitorio.

**Implementazione:**

Crea `src/lib/retry.ts`:
```typescript
export async function withRetry<T>(
  fn: () => Promise<T>,
  { retries = 2, delayMs = 500 }: { retries?: number; delayMs?: number } = {}
): Promise<T> {
  let lastError: unknown
  for (let i = 0; i <= retries; i++) {
    try {
      return await fn()
    } catch (err) {
      lastError = err
      if (i < retries) await new Promise(r => setTimeout(r, delayMs * (i + 1)))
    }
  }
  throw lastError
}
```

Modificare `src/hooks/useSession.ts`:
- Wrappare `supabase.rpc('record_consent_action', ...)` con `withRetry(..., { retries: 2, delayMs: 500 })`
- Mostrare un indicatore visivo "Riprovando..." durante i retry
