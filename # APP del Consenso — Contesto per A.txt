# APP del Consenso — Contesto per Antigravity

## Cos'è l'app

**APP del Consenso** è una Progressive Web App (PWA) mobile-first che permette a due adulti di documentare reciprocamente il proprio consenso consapevole in tempo reale. Il flusso avviene tramite pairing di prossimità (QR code o codice numerico), conferma condivisa con press-and-hold, e revoca immediata con propagazione realtime all'altro partecipante.

**Principi cardine:**
- Privacy by design: nessun audio, documento d'identità o metadato superfluo
- Reciprocità: entrambi i partecipanti devono confermare attivamente
- Revocabilità: chiunque può revocare in qualsiasi momento durante la sessione
- Pseudonimia: gli utenti sono identificati solo da un display name scelto al signup

L'app **non ha valore legale probatorio** — ha valore documentale/comunicativo.

## Database schema (sintesi)

### Tabelle principali

**`profiles`** — identità utente (sincronizzata con Clerk)
- `id TEXT PRIMARY KEY` (Clerk user ID, non UUID)
- `display_name TEXT` (2-30 chars)
- `avatar_color TEXT` (hex)

**`pairing_codes`** — codici temporanei di pairing
- `code TEXT UNIQUE` (6 cifre, regex `^\d{6}$`)
- `expires_at TIMESTAMPTZ` (10 minuti dall'emissione)
- `used_at TIMESTAMPTZ` (NULL = disponibile)

**`consent_sessions`** — sessioni di consenso
- `status TEXT` (pending → active → confirmed | revoked | expired)
- `expires_at TIMESTAMPTZ` (5 ore dall'inizio)
- `integrity_hash TEXT` (SHA-256 partecipanti + timestamp)
- `participant_count INT`
- `revoked_by TEXT` (FK a profiles.id)

**`consent_participants`** — partecipanti per sessione
- `role TEXT` (initiator | participant)
- UNIQUE(session_id, user_id)

**`consent_confirmations`** — audit trail immutabile
- `action TEXT` (confirmed | revoked)
- `action_hash TEXT` (SHA-256 per integrità)

**`v_current_consent_status`** — view: ultimo stato per (session_id, user_id)

### RPC Functions disponibili (SECURITY DEFINER)
- `upsert_my_profile(display_name, avatar_color)` → void
- `create_consent_session(participant_ids[], integrity_hash)` → session_id
- `create_session_from_pairing_code(code, integrity_hash)` → session_id
- `record_consent_action(session_id, action, action_hash)` → void
- `get_session_profiles(session_id)` → rows con display_name + avatar_color
- `get_session_metrics(session_id)` → {confirmed_count, any_revoked}
- `generate_pairing_code()` → TEXT (6 cifre univoche)
- `expire_stale_sessions()` → INT (count scadute)

## Autenticazione (Clerk + Supabase)

Il sistema usa **Clerk** come identity provider e **Supabase** come database.

**Flusso:**
1. Utente si autentica su Clerk → ottiene JWT Clerk
2. `setSupabaseAccessTokenGetter()` in `src/lib/supabase.ts` inietta il token Clerk nelle richieste Supabase
3. RLS policies usano `requesting_user_id()` (estratto dal JWT Clerk) per autorizzare
4. Tutti i `profiles.id` sono Clerk user IDs (TEXT, non UUID)

**File chiave auth:**
- `src/lib/supabase.ts` — client con dynamic token getter
- `src/hooks/useAuth.ts` — `useAuthInit()` bridge Clerk→Supabase + fetch profilo
- `src/stores/authStore.ts` — Zustand store: `{ user, profile, loading }`
- `src/main.tsx` — ClerkProvider con redirect URLs
